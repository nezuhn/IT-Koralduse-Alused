# Kokkuvõte ja refleksioon

## Mis on Git ja miks seda kasutatakse?

Git on versioonihaldussüsteem, mis võimaldab jälgida failide muudatusi ajas. Seda kasutatakse peamiselt tarkvaraarenduses, kuid sobib ka dokumentatsiooni haldamiseks. Git võimaldab mitmel inimesel korraga sama projekti kallal töötada ilma üksteise tööd üle kirjutamata.

## Mis vahe on commitil ja pushil?

Commit on nagu foto sinu failidest kindlal ajahetkel – see salvestab muudatused lokaalselt sinu arvutisse. Push seevastu saadab need kommitid GitHubi serverisse. Mõtle nii: commit on päeviku sissekanne, push on selle päeviku jagamine teistega.

## Miks on harud (branches) kasulikud?

Harud võimaldavad teha katsetusi ja arendusi ilma põhitööd riskimata. Kui oled harus midagi ära rikkunud, saab selle lihtsalt unustada ja põhiharu (main) jääb puhtaks. Harusid kasutades saab mitu arendajat korraga erinevaid funktsioone arendada.

## Mis on Pull Request ja milleks seda vaja on?

Pull Request on palve ühendada ühe haru muudatused teise haruga. See on koht, kus meeskond saab enne ühendamist arutada, koodi üle vaadata ja parandusi paluda. PR on nagu kvaliteedikontroll – veendutakse, et muudatused on korralikud enne, kui need põhiharusse jõuavad.

## Mille poolest erineb Git kõigest muust (nt failide kopeerimisest USB-le)?

Failide kopeerimisel USB-le kaob ajalugu – sa tead ainult viimast versiooni, aga mitte seda, mida ja millal muudeti. Git säilitab kogu ajaloo koos autorite ja kuupäevadega. Giti abil saad võrrelda kahte versiooni, pöörduda tagasi vana versiooni juurde või isegi paralleelselt arendada.

## Kuidas seostub Git projektijuhtimisega?

GitHubi Issues on nagu ülesannete nimekiri – iga probleem, idee või puudujääk saab oma numbri ja arutelu. Projekti kaudu (Projects tab) saab neid Issueseid liigutada veergudes (nt "Teha", "Teen", "Valmis") nagu Kanban-tahvlil. See muudab Giti mitte ainult versioonihalduseks, vaid ka projektijuhtimise tööriistaks.

## Kas hakkaksin GitHubi edaspidi kasutama? Milleks?

Jah, kindlasti. Plaanin GitHubi kasutada kõikide kooliprojektide jaoks, et harjuda versioonihaldusega. Samuti saan sinna panna oma isiklikke projekte nagu retseptide kogu või õppematerjalid. Hiljem saan näidata oma GitHubi profiili potentsiaalsele tööandjale – see näitab, et mõistan kaasaegseid tööriistu.

## Mis oli selle töö juures kõige raskem ja kõige lihtsam?

Kõige lihtsam oli README failide täitmine tekstiga, sest Markdown on loogiline ja lihtne. Kõige raskem oli meeles pidada alati kirjeldavaid commit-sõnumeid kirjutada – harjumus kirjutada "fix" või "update" on tugev. Samuti võttis aega aru saada, kuidas Pull Requestid töötavad, aga kui ühe korra ära tegin, sain asjast pihta.

Kokkuvõttes on Git ja GitHub väga võimsad tööriistad. Alguses tundub palju uusi mõisteid, aga praktikas harjub kiiresti. Olen kindel, et need oskused tulevad mulle IT karjääris kasuks.
